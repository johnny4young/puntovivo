# Accessibility (WCAG 2.2 AA)

> Status: shipped engine (axe-core helper + contrast and adaptive-mode gates +
> representative browser sweep).

This doc explains how Puntovivo enforces accessibility in CI and
local release validation, and what to do when a new component fails a
gate. Same principle as the coverage floor and perf budgets: every
regression is a deliberate choice, documented in the same PR that
produces it.

## What is enforced today

| Surface                                                                                                 | Where                  | Gate runner                                                                              |
| ------------------------------------------------------------------------------------------------------- | ---------------------- | ---------------------------------------------------------------------------------------- |
| Component-level axe-core WCAG 2 A + AA ruleset                                                          | Vitest                 | `assertNoA11yViolations(container)` from `apps/web/src/test/a11y.ts`                     |
| Color token contrast (`background/foreground`, `card/card-foreground`, `primary/primary-foreground`, …) | `ci:web`               | `scripts/check-contrast.mjs` after the bundle-size gate                                  |
| Reduced motion and Windows forced-colors recipes                                                        | `ci:web`               | `scripts/check-adaptive-ui.mjs` against the shared Operator Deck CSS                     |
| Playwright a11y smoke on the top user-facing routes                                                     | Local E2E release gate | `runAxeOnPage(page)` from `e2e/web/support/a11y.ts`, exercised by `e2e/web/a11y.spec.ts` |

The full keyboard-only `/sales` smoke is shipped. Playwright also pins the
screen-reader contract for the responsive admin navigation and cashier payment
drawer, a 200%-equivalent checkout width, reduced motion, and simulated Windows
forced colors. The remaining platform-specific follow-up is a real Windows
NVDA sweep. The `aria-keyshortcuts` declarations
aligned with the shortcut map already landed inline with
slice A on `SalesCheckoutPanel` (Charge `F1`, Suspend `Mod+P`, Toggle
Suspended `Mod+R`).

This guide is the canonical accessibility reference for engineering,
pre-sales, auditors, and non-technical reviewers.

## Why accessibility matters

Two reasons:

1. **Legal**: some LATAM accessibility regulations (Ley 1346 in
   Colombia, NMX-R-050 in Mexico) require WCAG conformance for
   commercial software. Enterprise pilots ask for the certification
   before signing.
2. **Operational**: cashiers spend their day in this product. A
   keyboard-only operator with a wrist injury or a screen-reader
   operator with low vision should be able to ring a sale without
   workarounds.

## How the gates run

### axe-core in component tests

`apps/web/src/test/a11y.ts` exports `assertNoA11yViolations(container, options?)`.
It runs `axe-core` against the rendered subtree with the WCAG 2 A +
AA ruleset and throws on any violation at or above the configured
severity floor (default `serious`, opt in to `moderate` for critical
surfaces).

Use it in any component test that exercises form fields, dialogs,
icon-only buttons, or interactive widgets:

```ts
import { assertNoA11yViolations } from '@/test/a11y';

it('passes axe-core WCAG 2 AA on the happy-render path', async () => {
  const { container } = render(<MyComponent />);
  await assertNoA11yViolations(container);
});
```

The helper is opt-in — existing tests stay green until they pick it
up. Add the call whenever you touch a new UI surface.

To raise the bar for a critical screen (checkout, login, settings):

```ts
await assertNoA11yViolations(container, { severityFloor: 'moderate' });
```

### Contrast gate

`pnpm run ci:web` chains:

1. `pnpm audit --prod`, typecheck, lint, unit tests with coverage, build.
2. `node --test scripts/check-bundle-size.test.mjs scripts/check-contrast.test.mjs`.
3. `node scripts/check-bundle-size.mjs` ().
4. `node scripts/check-contrast.mjs` — parses every concrete CSS
   scope in `apps/web/src/styles/theme.css`, converts each
   `oklch(L C H)` value to linear sRGB, computes the WCAG relative
   luminance + contrast ratio for the enforced pairs, and fails
   when any pair falls under its floor.

The enforced pairs are:

- `background` / `foreground`
- `card` / `card-foreground`
- `popover` / `popover-foreground`
- `muted` / `muted-foreground`
- `accent` / `accent-foreground`
- `primary` / `primary-foreground`
- `secondary` / `secondary-foreground`
- `destructive` / `destructive-foreground`
- `background` / `muted-foreground` (cross-surface — slice B)
- `card` / `muted-foreground` (cross-surface — slice B)
- `popover` / `muted-foreground` (cross-surface — slice B)
- `warning-50` / `warning-700` (badge-warning surface — )

Default floor is **4.5:1** (WCAG AA body text). Puntovivo's shared
button styles use `text-sm font-semibold`, so the `primary` and
`destructive` pairs also use the body-text floor rather than the
relaxed large-text rule.

Tolerant paths:

- Pairs with one side missing from a scope emit a warning, not a
  fail (a typography utility class like `.display` doesn't need
  color tokens).
- Non-oklch values emit a warning (parser only covers oklch for v1;
  hsl support can land in a follow-up).
- Scopes with zero relevant declarations are skipped — they belong
  to typography utility classes, not to a theme variant.

## How to fix a contrast regression

1. Open `apps/web/src/styles/theme.css`. Find the offending
   declaration in the `:root` or `.dark` scope.
2. Adjust the `oklch(L C H)` so the contrast hits the floor:
   - For body text pairs, bump the foreground darker (lower L) or
     the background lighter (higher L) until the ratio is ≥ 4.5.
   - Verify locally with `node scripts/check-contrast.mjs` — the
     PASS output prints the resulting ratio.
3. Commit the change in the PR that produced the regression. The
   gate stays green.

If the regression is intentional and you legitimately need to lower
the floor (e.g. a new button variant that only renders large text),
update `PAIR_FLOOR_OVERRIDES` in `scripts/check-contrast.mjs` with a
comment explaining why.

## How to silence a single axe violation

Sometimes a violation is a false positive (axe sees a `role` it does
not expect; the surface is intentionally not interactive). Two
options:

1. **Scope the assertion** to a known-good subtree:

   ```ts
   await assertNoA11yViolations(container.querySelector('[data-testid="happy-path"]')!);
   ```

2. **Configure the run** to exclude the offending rule with an
   explicit comment:

   ```ts
   await assertNoA11yViolations(container, {
     runOptions: {
       runOnly: { type: 'tag', values: ['wcag2a', 'wcag2aa'] },
       rules: {
         // The role here is intentional — see ADR-0009.
         'aria-allowed-role': { enabled: false },
       },
     },
   });
   ```

The comment is mandatory: a reviewer must be able to tell why the
suppression is legitimate without paging the original author.

## How to add a new pair to the contrast gate

1. Open `scripts/check-contrast.mjs` and append the pair to
   `ENFORCED_PAIRS`.
2. If the pair backs large text only, add an entry to
   `PAIR_FLOOR_OVERRIDES` with the 3:1 floor.
3. Run `node scripts/check-contrast.mjs` to confirm the pair passes.

## ARIA conventions in this codebase

A few patterns reviewers should check on every PR that touches the
UI:

- **Icon-only buttons** must declare `aria-label` (or `aria-labelledby`
  pointing at a visible label). The `Bell`, `User`, `Menu` etc.
  buttons in `Header.tsx` and `Sidebar.tsx` follow this pattern.
- **Disclosure widgets** (collapsible sections, menus, popovers)
  declare `aria-expanded` plus `aria-controls` pointing at the
  region they reveal. Section disclosure in `Sidebar.tsx` is the
  reference.
- **Menus opened from a trigger button** declare `aria-haspopup="menu"`
  on the trigger.
- **Modals** rely on the shared `Modal` component which already
  implements focus trap + ESC handling + return-focus to the
  opener. New dialogs should reuse it instead of rolling their own.
- **Status colors** must be paired with a text or icon affordance so
  the meaning is visible without color (`status pill` with text
  label, `KdsOrderCard` with timer label, …).


Failure modes the smoke reports clearly:

- An axe violation at or above `serious` — the error message lists
  every offending rule, the target selector, the WCAG help URL, and
  the failure summary axe produced.
- A console error or 4xx / 5xx network response while the page
  rendered — the `ClientIssueTracker` collects it and
  `expectNoClientIssues` asserts the list is empty.

### How to fix a regression

1. Reproduce locally: `pnpm run test:e2e:web -- a11y.spec.ts`.
2. Open the failure summary — axe points at the exact selector that
   failed (`button:not([aria-label])`, `label[for]=missing`, …).
3. Fix the source file the selector resolves to. The most common
   culprits are: icon-only `<button>` missing `aria-label`, a
   `<select>` rendered as a custom popover without `aria-haspopup`,
   a heading hierarchy skip (`<h1>` straight to `<h3>`), an `<input>`
   without `<label htmlFor>` or `aria-labelledby`.
4. Re-run the smoke. The trace under `playwright-report/web/`
   contains a per-test axe scan snapshot that helps when the failure
   is intermittent.

### How to suppress a single rule

Same contract as the component helper: every suppression carries a
written justification a reviewer can act on. Example:

```ts
await runAxeOnPage(page, {
  extraTags: [],
  // The role on the data-table header here is intentional; see
  // ADR-0009 for the reason we ship a custom grid.
  exclude: ['[data-testid="custom-grid-header"]'],
});
```

Use `exclude` for surface-level suppressions and the component-level
`runOptions.rules` (in `assertNoA11yViolations`) for rule-level
suppressions. Never silence a rule globally.

## Tabular row keyboard navigation

A non-trivial accessibility gap that lived for months in the
codebase: data tables rendered `<tr>` rows with `onClick` only —
no `tabIndex`, no `onKeyDown`, no focus management. A cashier
using only keyboard could not select a product, customer, or sale
from any table; the path required mouse.

`ProductSearchDialog` is the first table to close the gap. The
pattern is intentionally light so it stays applicable to every
data table without changing semantics:

1. **Preserve table semantics**. Do NOT add `role="listbox"` to
   `<tbody>` or `role="option"` to `<tr>`. Tables of SKU + Name +
   Stock + Price + Unit are real columnar data — the implicit
   `role="row"` (which already accepts `aria-selected`) plus a
   roving tabindex is the correct WAI-ARIA pattern.

2. **Roving tabindex**. Maintain an `activeRowIndex` state (default
   `0`); each row renders `tabIndex={index === activeRowIndex ? 0 : -1}`.
   Only the active row enters the Tab order — the Modal focus trap
   (whose focusable query includes
   `[tabindex]:not([tabindex="-1"])`) skips inactive rows
   automatically without extra wiring.

3. **`aria-selected` reflects the application state**, not the
   roving focus. The active row may not be the selected row (the
   cashier might be navigating through results before pressing
   Enter on one).

4. **Reset on items change with the React 19 same-render setState
   pattern**, NOT `useEffect`:

   ```tsx
   const [activeRowIndex, setActiveRowIndex] = useState(0);
   const [lastItemsKey, setLastItemsKey] = useState('');
   const itemsKey = items.map(item => item.id).join('\u0000');

   if (itemsKey !== lastItemsKey) {
     setLastItemsKey(itemsKey);
     setActiveRowIndex(0);
   }
   ```

   The conditional setState during render queues a same-render
   re-render and avoids the lint rule
   `react-hooks/no-state-update-in-effect`, which forbids cascading
   renders. See
   [react.dev — You might not need an effect](https://react.dev/learn/you-might-not-need-an-effect)
   for the canonical pattern.

5. **Key bindings, mirroring CommandPalette**:
   - `ArrowDown` → `Math.min(index + 1, items.length - 1)` (no
     wrap-around — keeps muscle memory consistent across the
     renderer).
   - `ArrowUp` → `Math.max(index - 1, 0)` (no wrap).
   - `Home` → row `0`.
   - `End` → row `items.length - 1`.
   - `Enter` / `' '` (Space) → fire the same select callback the
     existing `onClick` fires. Mouse path stays untouched.
   - Every handled key calls `event.preventDefault()`. Unhandled
     keys bubble so document-level shortcuts (`Alt+U` to focus the
     unit select, etc.) keep working.

6. **`onFocus` syncs `activeRowIndex`** when a row receives focus by
   mouse click or programmatic focus. This keeps the roving
   tabindex in sync with whichever row the cashier is interacting
   with, regardless of input device.

7. **`focus-visible` styling, not `focus`**. The active row carries
   `focus:outline-none focus-visible:bg-primary-50/60 focus-visible:ring-2 focus-visible:ring-primary-500/60`.
   El anillo aparece bajo navegación de teclado pero no bajo mouse
   click, alineado con la convención de la plataforma.

### Applying the pattern to other tables

There are two surfaces in the codebase: tables that go through the
shared `DataTable` primitive (the vast majority — `SalesHistoryTable`,
`PurchasesHistoryTable`, `OrdersHistoryTable`,
`QuotationsHistoryTable`, `CustomersPage`, `ProductsPage`, plus the
`ResourcePage` wrapper that hosts the admin / settings lists), and
bespoke tables that render their own `<table>` (today only
`ProductSearchDialog`). Each surface has its own recipe.

#### Consumers of `DataTable` ()

`DataTable.tsx` already owns the full roving-tabindex contract
(ArrowUp / ArrowDown / Home / End plus the `<tr>` `tabIndex`,
`aria-selected`, `focus-visible` ring). What you need to wire is
the row-activate action — what `Enter` and `Space` should do on
the focused row:

```tsx
<DataTable
  columns={columns}
  data={items}
  // ...existing props
  onRowActivate={row => onView(row.id)}
/>
```

Pick the primary mouse action of the row (Eye / View, Pencil /
Edit, Open Details) and pass the same callback. The activate path
is unconditional — it does not require `enableRowSelection`, and
mouse `onClick` semantics stay intact.

Consumers that route through `ResourcePage` get the same prop —
it forwards `onRowActivate` to its embedded `DataTable`.

Role gating: when the primary action is role-restricted (e.g.
"Edit" only for manager / admin), gate the prop itself
(`onRowActivate={canManage ? handleOpenEdit : undefined}`) so the
keyboard path mirrors the column-level button availability.

Tests: add one regression case per consumer that focuses a row and
asserts the callback fires with the right id. Mirror the pattern in
`SalesHistoryTable.test.tsx`, `PurchasesHistoryTable.test.tsx`,
`OrdersHistoryTable.test.tsx`, `QuotationsHistoryTable.test.tsx`.

#### Bespoke `<table>` outside DataTable ()

If the surface renders its own `<table>` (today only
`ProductSearchDialog`), follow the inline pattern that
shipped: `activeRowIndex` + `lastItemsKey` state, `rowRefs`,
`onKeyDown` handler on each `<tr>`, etc. See
`apps/web/src/components/dialogs/ProductSearchDialog.tsx` for the
canonical implementation and
`ProductSearchDialog.test.tsx > keyboard navigation ()`
for the test pattern (~13 cases).

Prefer migrating the surface to `DataTable` when feasible — the
ProductSearchDialog kept its bespoke layout because of the
right-side selection panel, but most plain data tables can adopt
`DataTable` and inherit keyboard support for free.

## Responsive payment drawer

`SalePaymentModal` keeps its public name and form contract, but renders through
the shared `Drawer`. The drawer owns one labelled `role="dialog"`, traps focus,
closes with `Escape`, locks body scroll, and restores focus explicitly to
`#sales-product-search-input`. The payment title, total summary, scrollable form,
and action footer are four structural regions, so a screen-reader or keyboard
user does not lose the Confirm action when switching tender modes.

`e2e/web/payment-drawer-responsive.spec.ts` pins the settled geometry and focus
round-trip at 1440x900, 768x1024, 640x800, and 390x844. The 640 CSS-pixel case
represents the layout exposed by a 1280-pixel display at 200% browser zoom; it
does not replace native browser and operating-system zoom validation. The suite
checks that cash/card and split-tender transitions preserve the panel rectangle,
keep close, total, and confirmation controls reachable, use internal vertical
scrolling without horizontal overflow, and emit no client errors. Set
`PUNTOVIVO_AUDIT_DIR` when running the spec to capture local EN/ES screenshots
without adding binary artifacts to git.

## Mobile workspace navigation

Below the desktop `xl` breakpoint, the application renders one dedicated
navigation drawer instead of keeping the translated desktop rail in the
accessibility tree. The open drawer is a labelled modal dialog and inherits
the shared dialog contract: focus enters the close control, `Tab` stays inside,
`Escape` closes it, body scroll is locked, and focus returns to the header
opener.

The workspace selector is a single-select radio group with roving arrow-key,
Home, and End navigation. Only the selected workspace's routes are mounted, so
screen readers do not traverse hidden route inventories. Cashiers skip the
selector entirely because their role resolves to the single Sell workspace.
Desktop disclosure chevrons announce the action and workspace separately, for
example `Expand Catalog` / `Collapse Catalog`, instead of duplicating the
adjacent workspace-link name.

Permanent Playwright coverage lives in
`e2e/web/navigation-responsive.spec.ts` at 768×1024 and 390×844. It pins
admin selection, direct-route restoration, cashier gating, Escape close,
opener-focus restoration, no horizontal overflow, and zero client errors.

## Assistive-technology sweep

The shared dialog hook isolates every background sibling with both
`aria-hidden="true"` and `inert` while a modal is open. Isolation walks every
ancestor level, so it covers portal-backed drawers and inline navigation
dialogs. State is reference-counted and preserves prior attributes, which keeps
the application hidden while dialogs are stacked and restores it only after the
last owner closes. Focus moves into the active dialog before isolation and
returns to the opener (or an explicit restore target) on close.

The payment drawer exposes one concise modal to assistive technology:

- the total is an atomic, polite status named by its localized visible label;
- payment-method tiles form one labelled group and the form-library backing
  select is removed from the accessibility tree;
- validation and checkout failures are alerts, while split-tender mismatch is
  a polite status; and
- mobile action order matches DOM and focus order (`Cancel`, then
  `Confirm Sale`).

`e2e/web/assistive-technology.spec.ts` is the permanent semantic smoke. At a
390×844 viewport it validates the Spanish admin workspace dialog and the
English cashier payment drawer, including labels, live regions, background
isolation, arrow-key selection, Escape/focus restoration, action order, and
zero browser errors. At 1280×800 it additionally emulates reduced motion and
forced colors: checkout must open with no running shared transition, keep focus
inside the dialog, expose system-colored control boundaries and a visible focus
ring, and avoid horizontal overflow. With `PUNTOVIVO_AUDIT_DIR` set, the spec
stores screenshots and Playwright ARIA snapshots as local review evidence.
