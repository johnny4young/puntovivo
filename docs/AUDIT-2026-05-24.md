# Project Audit — 2026-05-24

Cross-cutting audit of the Puntovivo codebase covering security, frontend
performance, database / Drizzle, and code quality. The audit is the source
for tickets **ENG-166 … ENG-181** in `ROADMAP.md` §3b. Each ticket below
lists the concrete file:line findings the agent should resolve when the
ticket is picked up.

The audit was scoped by the operator with four explicit dimensions:

1. Security and vulnerability surface.
2. Performance — web, HTML, React.
3. Database, SQL and Drizzle reliability + future model.
4. Maintainability — clean code, dead code, naming, repeated patterns,
   documentation of critical logic.

Findings are reported as actionable bullets — fix locations, the
underlying problem, and the concrete change. Implementation is **not**
attempted in this PR; each cluster is a future ticket.

> Snapshot context: 943 TS/TSX files, ~229 769 lines across `apps/web`,

## Security

### ENG-166 — Critical security closure (2026-Q2/Q3 audit)

> **Status: Closed 2026-05-24.** All 11 bullets shipped as a single
> slice. Each item below is now pinned by a regression test under
> `packages/server/src/__tests__/` or `apps/desktop/src/main/__tests__/`.
> See [SECURITY.md §Critical security closure](./SECURITY.md) for the
> consolidated description of the new posture.

Bundles the highest-severity findings that can be closed without external
dependencies or schema migrations.

- **SSE client id uses `Math.random()`** —
  `packages/server/src/realtime/sse.ts:188` generates
  `sse_${Date.now()}_${Math.random().toString(36).substring(2, 9)}`. Both
  components are predictable; an attacker that can guess a recent id
  could attempt to hijack the channel. Replace with
  `randomBytes(16).toString('hex')` from `node:crypto`.
- **No `@fastify/helmet` / security headers** — confirmed via
  `grep -rn "helmet\|Content-Security-Policy\|X-Frame-Options"` returning
  zero hits across `packages/server/src` and `apps/desktop/src/main`. Add
  `@fastify/helmet` to `packages/server/src/index.ts` with a `contentSecurityPolicy`
  appropriate to the Electron renderer + web app (allow `self`, the Vite
  proxy, fonts.googleapis.com, fonts.gstatic.com; block inline scripts in
  production), plus `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`,
  and `X-Content-Type-Options: nosniff`. Mirror the CSP in
  `apps/web/index.html` (a `<meta http-equiv="Content-Security-Policy">`
  tag for static hosting) and in the Electron `webContents`
  `onHeadersReceived` hook so renderer requests carry the same policy.
- **`sameSite: 'lax'` on refresh + realtime cookies** —
  `packages/server/src/security/authTokens.ts:169` and
  `packages/server/src/trpc/routers/auth.ts:62,76`. Promote to
  `sameSite: 'strict'` after confirming the embedded Electron flow does
  not cross top-level navigations. If a downgrade is required for the
  web build only, branch the cookie attributes by `request.host`.
- **Login timing-attack on user enumeration** —
  `packages/server/src/trpc/routers/auth.ts:116-123` short-circuits when
  the user does not exist, skipping the Argon2 verify. Even with the
  shared error message, the response time is materially different. Hash
  a constant dummy password with `argon2.verify(DUMMY_HASH, input.password)`
  on the not-found branch to equalise timings.
- **Argon2 default params drift risk** — every call to `argon2.hash` /
  `argon2.verify` in `auth.ts:136,358,368` and `users.ts:79,267` uses the
  library default. Wrap them in a `hashPasswordSecurely()` /
  `verifyPasswordSecurely()` helper in `security/passwords.ts` that pins
  `type: argon2.argon2id, memoryCost: 65_536, timeCost: 3, parallelism: 4`
  (OWASP 2025 minimum), plus a `needsRehash` helper for incremental
  upgrades.
- **`/api/trpc/*` not rate-limited at the procedure layer** — confirmed
  via `grep`. The `@fastify/rate-limit` registration in ENG-025 covered
  the HTTP path but not per-procedure buckets. Add `tRPC` middleware on
  `auth.refresh`, `auth.changePassword`, `auth.resetPassword`,
  `users.create`, `users.resetPassword`, and `desktopSession.register`
  with stricter caps than the per-IP shell. See ENG-165 for the full
  bucket plan; ENG-166 closes only the auth-critical subset.
- **Zod schemas omit `.strict()`** — every input schema under
  `packages/server/src/trpc/schemas/` uses default object behavior
  (`strip`). Explicitly add `.strict()` (or document the chosen contract)
  on every auth, users, sales mutation, and payments schema; tests should
  fail when extra keys are submitted.
- **Email not normalised before query or insert** —
  `packages/server/src/trpc/routers/users.ts:67-72` queries by raw
  `input.email`. Two operators can register `Admin@x.com` and
  `admin@x.com` as different accounts; SSO + IdP mappings later
  break. Normalise via `email.trim().toLowerCase()` in the Zod
  `transform` on every schema that accepts an email (auth login,
  users.create, users.update, passwordReset).
- **`/api/trpc` IP rate-limit uses `ctx.req.ip` without `trustProxy`** —
  `auth.ts:105`. When deployed behind nginx / Cloudflare the rate-limit
  key collapses to the proxy IP. Enable `trustProxy` only for the
  reverse-proxied `site_hub` deployment shape (or accept a configured
  CIDR list in env) and document the deployment expectation in
  `docs/SECURITY.md`. Pair with a tenant + user-scoped bucket (delegated
  to ENG-165).
- **Print HTML not sanitised before render** —
  `apps/desktop/src/main/index.ts:2146-2156` (`print-receipt` handler)
  validates `receiptHtml` is a non-empty string, but the renderer
  ultimately loads the body via `data:text/html;...`. Add a server-side
  sanitiser (e.g. the same `DOMPurify` already in dependencies) over the
  HTML before the Electron main process opens the print window.
- **`PUNTOVIVO_OPEN_DEVTOOLS` env var opens devtools in any build** —
  `apps/desktop/src/main/index.ts:1707-1708`. Gate the open on
  `app.isPackaged === false` (or `process.env.NODE_ENV !== 'production'`)
  so an accidental staging deploy can never expose the DevTools surface.

**Acceptance:** every bullet above has a regression test pinning the new
behaviour (response header asserts, sanitiser unit tests, timing-equality
mock test, Argon2 wrapper params test, email normalisation test); the
Q2/Q3 audit doc updates each item to "closed".

**Closure (2026-05-24)**: `ENG-166` shipped. The seven new server test
files (`security-helmet`, `security-passwords`, `auth-timing-equality`,
`auth-email-normalisation`, `schema-strict`, `procedure-rate-limit`,
plus the extended `realtime-sse`) and the new desktop test
(`print-html-sanitizer` under `node --test`) gate the new behaviour
in CI. Per-procedure rate-limit closes the auth-critical subset; full
tRPC bucket plan remains as `ENG-165`. Token + session lifecycle full
treatment remains as `ENG-168` per the audit's own carve-out.

### ENG-167 — SQLite encryption at rest

> **Status update 2026-05-25 — Partial (Step-1).** Library swap + key
> plumbing landed. `better-sqlite3` now resolves to
> `better-sqlite3-multiple-ciphers@^12.10.0` (npm alias declared at
> root + server workspace); `DatabaseOptions.encryptionKey` +
> `ServerOptions.encryptionKey` route a 64-char hex key into
> `initDatabase`, which issues `PRAGMA cipher='sqlcipher'`,
> `PRAGMA legacy=4`, and `PRAGMA key` before the ENG-174 cluster (skipped on
> `:memory:`); `apps/desktop/src/main/db-key-store.ts` mints + seals
> the per-install 32-byte key via Electron `safeStorage` at
> `<userData>/data/.dbkey.enc` and aborts boot when
> `isEncryptionAvailable()` is false. Threat-model section landed in
> [`SECURITY.md` §"Database encryption at rest (ENG-167 Step-1)"](./SECURITY.md);
> open-path map in [`ARCHITECTURE.md` §"Database open path +
> encryption (ENG-167)"](./ARCHITECTURE.md). Regression suites pin
> every branch
> (`packages/server/src/__tests__/db-encryption.test.ts` +
> `apps/desktop/src/main/__tests__/db-key-store.test.ts`).
> `scripts/ensure-native-runtime.mjs` cache key now folds in
> `packageJson.name`, so swapping back to plain better-sqlite3 (or
> bumping the fork's major) invalidates the cached binary
> automatically. **Remaining for ENG-167b**: one-shot migration of
> pre-encryption cleartext DBs on first boot; restore-from-different-device
> key-prompt UX in `backup-bundle.ts`; cross-OS validation via the
> manual `build-desktop.yml` matrix (Linux + macOS + Windows). Step-1
> ships safely to dev (standalone keeps a cleartext fallback); rolling
> Step-1 alone to existing prod installs would break them — production
> deployment is gated on ENG-167b.

The local DB at `app.getPath('userData')/data/local.db` carries every
sale, customer, fiscal document, and Argon2 hash for the tenant. A stolen
laptop, a misconfigured backup, or a malicious peripheral on a shared
register exposes the full file. Confirmed via
`apps/desktop/src/main/index.ts:110` + `packages/server/src/db/index.ts`
where the DB is opened with the unencrypted `better-sqlite3` constructor.

- Evaluate `@journeyapps/sqlcipher` (drop-in for `better-sqlite3` with
  `PRAGMA key`) vs `node-sqlite3` SQLCipher build vs application-layer
  cryptography on backup only.
- Document the threat model in `docs/SECURITY.md`: encryption at rest
  protects against device theft and side-channel disk access; it does
  **not** protect against a running process with the unlocked key. The
  key must come from the OS keychain (macOS Keychain, Windows DPAPI,
  libsecret) using the existing Electron `safeStorage` API.
- Migration path: ship a one-shot encryption step on first launch of the
  upgraded build. New installs receive an encrypted DB by default. The
  Electron rebuild matrix (ENG-005) must be re-validated since SQLCipher
  ships its own native binary.
- Backup tooling (`apps/desktop/src/main/backup/`) must store encrypted
  bundles; restore prompts for the matching key when the source device
  differs.

**Acceptance:** packaged desktop build ships SQLCipher; the local DB is
unreadable with the standard sqlite3 CLI without the key; backup +
restore round-trip succeeds; ENG-005 desktop CI matrix stays green on all
three OSes; key resides only in the OS keychain.

### ENG-168 — Token and session lifecycle hardening

A cluster of session-management gaps surfaced during the audit; none are
critical in isolation but together they erode the auth invariants from
ENG-008b and ENG-025.

- **Realtime SSE token TTL of 120 s without refresh** —
  `packages/server/src/security/authTokens.ts:8` and
  `packages/server/src/realtime/sse.ts:235`. SSE connections live for
  hours; once the 120 s expires, the connection is no longer auditable.
  Either extend the TTL to the access-token lifetime and rotate at the
  same cadence, or push a `token-expiring` event 30 s before expiry that
  the renderer answers with a reconnect using a fresh token.
- **No device-revocation endpoint** — pairing flow in
  `packages/server/src/services/devices/authority.ts:183-230` issues
  device ids but nothing revokes them. Add `devices.revoke({ deviceId })`
  scoped by tenant + admin role; invalidating the device bumps
  `users.sessionVersion` for every user holding tokens bound to that
  device (or, if device-level binding is not granular enough yet,
  introduce a `devices.revoked_at` column read by `verifyAccessToken`).
- **Logout does not clear `desktopSession` in Electron main** —
  `auth.ts:194-200` increments `users.sessionVersion` server-side but
  `apps/desktop/src/main/session/desktopSession.ts` (introduced in
  ENG-025) retains the prior `tenantId`. The `db:*` / `sync:*` handlers
  then operate against a stale tenant until the renderer reloads. Wire
  the renderer to invoke a new `session:clear` IPC during the
  `auth.logout` mutation `onSuccess` callback, and have the main process
  call `desktopSession.clear()` synchronously.
- **No periodic cleanup of expired login attempts / sessions** — the
  ENG-008b table grows monotonically. Add a 24 h cleanup job in
  `packages/server/src/lifecycle/` that prunes
  `login_attempts WHERE expires_at < now()`, plus an audit row per run.
- **Pairing code claim has no audit row** — `authority.ts` records
  `createdByUserId` but the `claim` call writes nothing identifying the
  device / IP / user-agent that consumed the code. Add a
  `device.pairing.claimed` audit action with metadata for forensics.

**Acceptance:** SSE connections survive >24 h with continuous refresh;
revoking a device disables every token bound to it on the next request;
post-logout `db:*` IPC calls fail with `UNAUTHORIZED` (regression test);
expired login attempts purged after 24 h; pairing claim writes an audit
row.

### ENG-169 — Input boundary hygiene

Lower-severity but consistent gaps in input validation at the system
boundary.

- **Restore ZIP not allowlisted** — `apps/desktop/src/main/index.ts:1742-1768`
  (`handleRestoreDatabaseBackup`) accepts any ZIP the user picks. Enforce
  an allowlist of expected paths (`local.db`, `device-id`,
  `manifest.json`) and reject any entry containing `..` or absolute
  paths. Pair with a manifest signature check if the backup originated
  from another device (deferred until ENG-167 ships per-device keys).
- **`logoDataUrl` / `imageUrl` accept arbitrary URL scheme** — the
  resolved blocklist from ENG-025 (`packages/server/src/lib/urlSafety.ts`)
  only fires when the scheme part of the URL is normalised cleanly.
  Re-audit every Zod schema that accepts an image URL (`companies`,
  `receipt_templates`, `users.avatar` if added) and call
  `assertSafeImageUrl()` in the schema-level `refine` rather than at
  render time.
- **Whitespace and case not stripped on emails** —
  `packages/server/src/trpc/schemas/users.ts:14` does not trim or
  lowercase before validation. Add a Zod `transform` per ENG-166.
- **No `verbose` mode guard rails in production** —
  `packages/server/src/index.ts:406-408` logs `requestId` and `deviceId`
  in `verbose: true`. Add a startup check that refuses to boot with
  `verbose === true && NODE_ENV === 'production'` unless an explicit
  `PUNTOVIVO_ALLOW_VERBOSE_PROD=1` override is set, and log a warning
  when the override is used.

**Acceptance:** the four bullets above have regression tests; the
Q2/Q3 audit doc updates each item to "closed".

## Frontend performance

### ENG-170 — Web bundle optimisation

- **Sourcemaps shipped in production** — `apps/web/vite.config.ts:26`,
  `apps/desktop/vite.main.config.ts:6`,
  `apps/desktop/vite.preload.config.ts:6` all set `sourcemap: true`. The
  generated `.js.map` doubles the served payload on every chunk. Switch
  to `sourcemap: process.env.NODE_ENV !== 'production'` (or hidden
  sourcemaps + upload to the future error-tracking endpoint) and confirm
  the per-route `perf-budget.json` budgets shrink accordingly.
- **i18n eager-loads 46 JSON files at boot** —
  `apps/web/src/i18n/index.ts:1-81` imports 23 namespaces × 2 locales
  statically. A cashier that only visits `/sales` still parses
  `fiscal.json`, `kds.json`, `modules.json`, `aiSettings.json`, etc.
  Move every non-bootstrap namespace to a dynamic backend
  (`i18next-http-backend` against `/locales/<lng>/<ns>.json`, with Vite's
  glob-import to materialise the per-namespace JSON as static assets);
  keep `common`, `auth`, `nav`, `errors` in the static bootstrap so the
  login screen renders without a fetch.
- **No `manualChunks` for heavy libs** —
  `apps/web/vite.config.ts` lacks an explicit `build.rollupOptions.output.manualChunks`.
  Heavy modules (`jspdf`, `jspdf-autotable`, `exceljs`, `jszip`,
  `codemirror`, `@codemirror/*`, `@lezer/highlight`, `@dnd-kit/*`,
  `recharts`) load wherever they are first imported. Add explicit
  chunks `pdf`, `xlsx`, `codemirror`, `dnd`, `charts` so the entry
  bundle stays lean and the chunks are reused across routes.
- **Lazy import is correct in `exportService.ts:319,416`** but the
  consuming components (`ProductsPage`, `InventoryPage`, `AuditLogsPage`,
  `TableExportActions`) lack a Suspense / "Generating…" affordance
  bound to the dynamic-import promise. Wrap the call in an explicit
  `useExportTask()` hook that surfaces `isPending` so the export button
  disables until the chunk is downloaded.

**Acceptance:** main entry chunk gzip shrinks below the
`perf-budget.json` ceiling; `pdf` / `xlsx` / `codemirror` chunks load
only on the routes that need them (verified via
`vite build --report` or `rollup-plugin-visualizer`); login bundle no
longer contains fiscal / KDS / AI namespaces (verified via the bundle
report).

### ENG-171 — React re-render and data-fetching hygiene

- **`TenantContext.Provider` recreates value on every render** —
  `apps/web/src/features/tenant/TenantProvider.tsx:43-56`. Wrap the
  context value in `useMemo` keyed on `[tenant, sites, currentSite,
  isLoadingSites, switchSite]`. `switchSite` itself needs to be wrapped
  in `useCallback` inside `useActiveSite` to keep the dependency stable.
- **`AuthProvider.login` / `logout` are inline lambdas** —
  `apps/web/src/features/auth/AuthProvider.tsx:235-363`. Promote both to
  `useCallback` so memoised consumers are not invalidated by unrelated
  re-renders. Same fix for `register`, `refreshSession`, and any other
  exposed function.
- **`refetchOnWindowFocus: true` on heavy queries** —
  `apps/web/src/components/layout/Sidebar.tsx:226`,
  `apps/web/src/features/modules/ModulesContext.tsx:76`, and the
  anomaly-detection card refetch on every tab focus despite a 5-minute
  `staleTime`. Flip to `false` and replace with an explicit
  `refetchInterval` only where the underlying data really needs
  polling (anomaly badge).
- **`SalesPage` fires 11 queries on mount without prefetch** —
  `apps/web/src/features/sales/SalesPage.tsx:237-274`. Add a
  `queryClient.prefetchQuery` block when the user hovers
  `nav.sales`, plus `placeholderData: keepPreviousData` on
  `sales.list`, `sales.summary`, `customers.list`, `cashSessions.getActive`
  so navigation does not blank the page. Same audit on InventoryPage and
  PurchasesPage which follow the same pattern.
- **Multiple context providers nested in `App.tsx:240-252`** without
  selector boundaries means an `AuthProvider` re-render cascades through
  `TenantProvider`, `ModulesProvider`, `LocaleProvider`,
  `CommandPaletteProvider`. Migrate `ModulesProvider` and
  `LocaleProvider` to Zustand stores (Zustand is already adopted via
  ENG-018b) and keep React Context only for `Auth` + `Tenant` (which
  benefit from React's render-tree subscriptions).

**Acceptance:** React DevTools Profiler records show the `SalesPage`
header does not re-render when the cart updates; window-focus refetch
disabled on the Sidebar + Modules queries; navigation from `/dashboard`
to `/sales` no longer shows a blank shell.

### ENG-172 — DataTable virtualisation + CLS polish

- **`DataTable` renders every row in the current page** —
  `apps/web/src/components/tables/DataTable.tsx:50-250+` uses TanStack's
  `getPaginationRowModel` but has no virtualisation. For 50-row pages
  (FiscalDocumentListPage hard-codes `limit=50` at
  `apps/web/src/features/fiscal/FiscalDocumentListPage.tsx:51-52`) the
  scroll smoothness on low-end devices drops noticeably. Adopt
  `@tanstack/react-virtual` behind a `virtualised?: boolean` prop that
  flips on automatically when `data.length > 30`, preserving the current
  API. Reuse the existing `data-row-id` (E2E) and aria attributes.
- **`FiscalDocumentListPage` has no real pagination** —
  same file, the query is fixed `limit=50, offset=0`. Add the standard
  DataTable pagination controls (`pageIndex` state + `cursor` or
  `offset` change handler) so tenants with >50 fiscal documents can
  browse history. The current admin only sees the first page.

### ENG-174 — SQLite PRAGMA tuning + WAL backup safety

> **Status: Closed 2026-05-24.** All three sub-changes shipped as a single
> slice. PRAGMA cluster pinned in `packages/server/src/db/index.ts`,
> WAL checkpoint added to `apps/desktop/src/main/backup/backup-bundle.ts`,
> and new `scripts/ensure-migrations-bundled.mjs` preflight guard wired
> in `apps/desktop/package.json`. See [SECURITY.md §SQLite tuning](./SECURITY.md)
> for the consolidated description.

Today only `journal_mode = WAL` and `foreign_keys = ON` are configured
in `packages/server/src/db/index.ts:95-97`. Two follow-ups:

- **PRAGMA cluster missing** — concurrent readers (HTTP, SSE, sync
  worker, hardware worker) routinely contend for the writer slot. Add
  `busy_timeout = 5000`, `cache_size = -64000` (≈64 MiB), `mmap_size =
  268435456` (256 MiB; SQLite docs recommend up to 1 GiB),
  `temp_store = MEMORY`, and `wal_autocheckpoint = 1000`. Validate the
  defaults against the Electron memory budget on a 4 GB device (the
  cheapest LATAM POS we target — see `PERF-BUDGETS.md`).
- **Backup path does not force WAL flush** —
  `apps/desktop/src/main/backup/backup-bundle.ts:11-24` calls
  `db.backup(tmpPath)` directly. The promise resolves once the bytes are
  on disk, but the WAL file can still hold writes that have not been
  checkpointed. Add `PRAGMA wal_checkpoint(FULL)` (or `TRUNCATE`) and
  `PRAGMA synchronous = FULL` before the backup call; the existing
  `integrity_check` is a post-condition, not a prevention.
- **No pre-build verification that `migrations/` is bundled** —
  `packages/server/src/db/index.ts:128-131` throws on missing journal at
  startup. The Electron Forge build does not assert the resources
  contain the migrations folder; a broken bundle ships and crashes on
  first boot. Add a guard in `scripts/ensure-electron-binary.mjs` (or a
  new `scripts/ensure-migrations-bundled.mjs`) that runs as part of
  `preflight:desktop`.

**Acceptance:** PRAGMA values are pinned in a unit test against the
production-config code path; backup + power-loss simulation produces a
valid restore (chaos harness from ENG-067 if available); a build with a
missing `migrations/` resources entry fails preflight with an actionable

### ENG-176 — Money model + multi-currency LATAM readiness

> **Status update 2026-05-25 — Partial (Step-a + Step-b: money precision).**
> ADR-0009 (`docs/architecture/0009-money-storage-and-validation.md`)
> records the decision to keep `real` columns + add storage-layer money
> CHECK invariants (option b in the audit). Migration
> `0035_eng176a_money_checks.sql` recreates 17 monetary tables with
> ~40 non-negative CHECKs covering totals, subtotals, taxes, costs,
> prices, tips, service charges, opening floats, credit limits, and
> refund amounts. Migration `0036_eng176a_rounding_precision.sql`
> restores the two-decimal precision CHECK after the application
> `roundMoney()` sweep normalized sales, quotations, cash sessions,
> purchases, orders, inventory, products, and customer credit-limit
> write boundaries. Defensive UPDATE preludes round any historical
> drift to two decimals before recreation. Regression suite at
> `packages/server/src/__tests__/db-money-checks.test.ts` pins the
> invariants. **Remaining**: (a) ENG-176b — currency_code +
> exchange_rate seam on transactional tables + CHECKs for
> fiscal_documents, fiscal_document_items,
> payment_outbox (skipped in Step-a due to Drizzle snapshot drift);
> (b) ENG-176c — fiscal_identification_types catalog rename + MX/PE/CL
> seeds + fiscal_documents.status enum expansion. Each remaining slice
> is captured in `BACKLOG.md`.

The codebase stores monetary values as `real` (SQLite IEEE-754) and
hard-codes COP in `paymentOutbox.currencyCode default='COP'`
(`schema.ts:2058`). Two issues blocking LATAM expansion (PLAN-V3,
ENG-156):

- **Precision** — `0.1 + 0.2 !== 0.3` in IEEE-754. The application
  rounds to 2 decimals at the boundary today but the storage layer
  cannot defend against a future writer that skips the rounding. Pick
  one of:
  - Migrate every money column to `integer` cents (`COP_minor_unit`),
    add a `formatMinorAmount(value, currency)` helper, and rewrite every
    formatter / Zod schema / receipt renderer.
  - Or keep `real` and pin a runtime invariant via Drizzle `check`
    constraints that the absolute value × 100 is an integer. SQLite
    does not enforce `CHECK` retroactively, so the migration would
    require a one-shot validation + cleanup pass.
- **Multi-currency missing on transactional tables** — `sales`,
  `sale_items`, `quotations`, `quotation_items`, `products`,
  `customers.creditLimit` do not carry a `currency_code`. ENG-156
  (sell-USD / settle-COP) cannot ship without it. Add `currency_code`
  (FK to `currency_catalog` from ENG-017), `exchange_rate_at_sale`, and
  `settle_currency_code` (nullable, only set when ≠ sell currency).
- **Identification-type catalog hard-coded to DIAN** —
  `schema.ts:3354-3369` defines `dian_identification_types`. SAT (MX)
  and SUNAT (PE) need RFC / CURP / RUC / DNI etc. Rename to a generic
  `fiscal_identification_types(code, country_code, abbr, nameEn, nameEs,
  natural_person)`; backfill the DIAN rows with `country_code = 'CO'`;
  add the SAT + SUNAT seeds. The fiscal adapter contracts must look up
  by `country_code`.
- **`fiscal_documents.status` enum is closed to DIAN states** —
  `schema.ts:3404-3410` (`pending`, `sent`, `accepted`, `rejected`,
  `contingency`). SAT, SUNAT, SII, AFIP need additional states
  (`voided`, `notified_correction`, `partial_send`, …). Either expand
  the enum or migrate to `text('status')` + a per-country lookup table.

This ticket is the data-model dependency that ENG-156 (multi-currency
operations) and ENG-161 (NFe Brazil foundations) inherit. It is sized to
be implementable in one sprint **without** waiting for the LATAM
expansion tickets to land; the schema preparation is forward-only.

**Acceptance:** money columns either fully migrated to integer minor
units or pinned by a CHECK invariant; `currency_code` present and
non-null on every transactional row, defaulting to the tenant locale;
`fiscal_identification_types` covers CO + MX + PE + CL; status enum
extends to at least the SAT and SUNAT state set; existing tests stay
green; new tests cover round-tripping a USD-sold / COP-settled sale.

### ENG-177 — Optimistic versioning + soft-delete policy

- **No `version` column on mutable catalogs** — `users` carries
  `sessionVersion`, but `products`, `customers`, `providers`,
  `categories`, `sequentials`, `tenant_locale_settings` have no
  optimistic-concurrency guard. Two operators editing a price from
  different tabs silently overwrite each other. Add an
  `version: integer().notNull().default(0)` column on every
  user-editable catalog, increment on every UPDATE, and require the
  client to pass the previous version on the next mutation
  (`tRPC` middleware throws `STALE_VERSION` when they diverge).
- **Soft-delete inconsistent** — catalogs use `is_active`; transactions
  use a `status` enum; neither captures "when" nor "by whom". Decide:
  introduce `deleted_at` + `deleted_by_user_id` on catalogs (preserving
  `is_active` as a derived view for the next release), or remove soft
  delete on the catalogs that do not need it. Document the policy in
  `docs/ARCHITECTURE.md` so the next data model addition follows the
  same rule.
- **`inventory_balances` seed is O(N) INSERTs** —
  `packages/server/src/services/inventory-balances.ts:81-120` loops
  `forEach` inserting one row per product. For a 50 k-product tenant
  the first access to a new site locks the DB >1 s. Replace with a
  single `db.insert(inventoryBalances).values(rows)` bulk; verify the
  upper-bound parameter count against better-sqlite3's
  `SQLITE_MAX_VARIABLE_NUMBER` (999 by default — chunk above that).
- **`sales` does not enforce a non-null `cashSessionId` invariant in
  the schema** — the `requireActiveCashSession` invariant lives in
  application code. Add a `CHECK (cash_session_id IS NOT NULL OR status
  = 'draft')` once the migration shim verifies all historical rows
  comply.

**Acceptance:** concurrent edit from two browsers raises
`STALE_VERSION`; soft-delete policy doc landed; site-onboarding
benchmark improves >10× on the 50 k-product fixture; new check
constraint pinned by a unit test that bypasses the application layer.
