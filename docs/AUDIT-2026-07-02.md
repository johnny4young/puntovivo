## Fixes applied in this branch

### Desktop (Electron main process)

- **Session gate + payload shape guard on
  `peripherals:dispatch-local-escpos`** (`apps/desktop/src/main/index.ts`).
  This was the one stateful IPC handler without a `desktopSession` gate: a
  pre-login or compromised renderer could drive the main process to open a
  TCP socket and write attacker-chosen ESC/POS bytes. Blast radius was
  already bounded by the RFC-1918 + port-allowlist transport policy, but
  the handler now requires a registered session like every other actuator
  and rejects malformed payloads with a clean `INVALID_PAYLOAD` result.
- **Global `web-contents-created` deny-by-default policy**
  (`apps/desktop/src/main/index.ts`). `setWindowOpenHandler` +
  `will-navigate` + `will-attach-webview` are now attached to every
  webContents, current and future (print windows included), so an
  ephemeral window can never be navigated off the app even if its creator
  forgets to attach handlers. The mainWindow keeps its own policy with the
  in-app reload carve-out.
- **Session gate + bounds on `device:set-id`**
  (`apps/desktop/src/main/index.ts`). The device id is server-issued during
  login, which registers the desktop session first, so a pre-login
  renderer has no business persisting one. Rejection is non-fatal renderer
  side (localStorage stays authoritative).

### Web — double-submit protection on money paths (highest-value fix)

A held or repeated F1 / Enter fires `form.requestSubmit()` directly at the
payment form, bypassing the disabled footer button. Each
`useCriticalMutation.mutate()` call deliberately mints a fresh idempotency
envelope, so the server's idempotency cache does not dedupe the second
fire — the concrete failure was **two completed sales, double stock debit,
double cash-session takings** within one network round-trip window.

- `useSalePaymentModal.ts` — `isSaving` guard as the first line of the
  submit handler.
- `useSalesFlows.ts` — `handleCheckout` returns early while
  `createMutation`/`completeDraftMutation` are pending (defense in depth);
  `handleResumeFromPanel` returns early while `resumeMutation` is pending
  (double-click previously hydrated two workspaces pointing at the same
  `serverSaleId`, which could `completeDraft` the same sale twice).
- `SuspendedSalesPanel.tsx` — the Retomar row button now disables while a
  resume is in flight.
- `useSalesModals.ts` — the same `isPending` guard on the cash-session
  open / close / record-movement handlers (their modal forms are also
  Enter-submittable).

### Web — cache correctness

- **Touch POS now invalidates the full sale-completion set**
  (`PosTouchScreen.tsx`). It previously invalidated only
  `cashSessions.getActive` + `products.list`, leaving `/sales` history,
  sales summary, stock counts and the cash-movement report stale for the
  whole 5-minute `staleTime` window after a touch sale. The canonical
  picker list was extracted to `SALE_COMPLETION_INVALIDATIONS` in
  `lib/invalidateGroups.ts` and is now shared byte-for-byte with the
  desktop `finishSaleEpilogue`.

### Web — performance on cashier hot paths

- **`Intl.NumberFormat` cache in `formatCurrency`** (`lib/utils.ts`).
  Formatter construction is ~0.1–1 ms and the touch grid built one per
  tile per cart tick (≤200 tiles). Formatters are immutable; they are now
  cached per (locale, currency, decimals) with a defensive cap.
- **`React.memo` on `PosTouchProductGrid` + stable `handleAddToCart`**
  (`PosTouchProductGrid.tsx`, `PosTouchScreen.tsx`). Cart taps no longer
  re-render every product tile.
- **Cart undo history is no longer persisted**
  (`useCartWorkspaceStore.ts`). `partialize` previously serialized up to
  20 full item-array snapshots per workspace into localStorage on every
  cart keystroke — a synchronous multi-hundred-KB write on the
  highest-frequency input path. Undo is a session-level affordance (a
  rehydrated cart already starts with an empty baseline, same as resumed
  drafts).
- **200 ms debounce on product search** (`ProductSearchDialog.tsx`, new
  `hooks/useDebouncedValue.ts`). `useDeferredValue` only deprioritizes the
  render; every settled keystroke still became its own
  `products.search` request.

### Web — resilience and correctness hardening

- **Route-level error boundary** (`appRouteHelpers.tsx`,
  `AppErrorBoundary.tsx`). A render crash in one page previously unmounted
  the entire app (shell included) to the root fallback. `ShellRoute` now
  wraps the page slot in a `RouteErrorBoundary` with a per-route retry, so
  the navigation chrome and the rest of the POS survive.
- **Exponential backoff on the web auto-sync loop**
  (`hooks/useOfflineSync.ts`). On the web path, a push that reports
  success while `pendingCount` stays above zero (conflict rows, stalled
  batch) re-triggered immediately — a tight retry loop bounded only by
  network latency. Automatic re-triggers now back off 5 s → 5 min when no
  progress is made, and reset on progress.
- **Fast-cash effect stale-closure fix** (`useSalePaymentModal.ts`). The
  F2 fast-cash microtask now reads `applyFastCash` through a ref (same
  pattern as `useRealtimeChannel`), removing the `exhaustive-deps`
  suppression and the latent stale-`grandTotal` money bug a future
  deferral refactor would have triggered.
- **`ConfirmModal` localized defaults** (`Modal.tsx`). The fallback
  `Confirm Action` / `Confirm` / `Cancel` / `Loading...` strings were
  hardcoded English; they now resolve from the `common` namespace.
- **Duplicate-key fix in the shortcut chip row**
  (`SalesCheckoutPanel.tsx`). Two shortcuts formatting to the same key
  label collided on `key={keyLabel}`.

## Verified strengths (no action needed)

The codebase is in unusually good shape — several classic failure classes
were checked and found already handled, with regression tests pinning them:

- **Electron hardening**: sandbox + contextIsolation + nodeIntegration
  off pinned by test; fuses (`runAsNode` off, asar integrity, no
  NODE_OPTIONS); tenant identity held main-process-side
  (`desktopSession`), never trusted from the renderer; table/field
  allowlists + parameterized SQL on the IPC DB bridge; zip-slip-safe
  backup extraction; safeStorage-sealed SQLCipher key with crash-safe
  cleartext→encrypted migration; strict print-HTML sanitizer with its own
  locked CSP; DevTools and env-var key overrides hard-disabled in
  packaged builds.
- **Auth**: in-memory access token (never persisted), httpOnly rotating
  refresh cookie with CSRF double-submit, single-flight refresh, token
  type discrimination + DB-backed re-verification with `sessionVersion`,
  pinned argon2id with a pre-warmed dummy hash to equalize
  user-not-found timing, JWT-secret strength policy that refuses weak or
  placeholder secrets on `site_hub` boots.
- **Money math**: one EPSILON-corrected `roundMoney` mirrored
  byte-identically client-side; header/tender/change/cash-collected all
  re-rounded at the source with documented invariants; discount >
  total rejected before any write.
- **Idempotency**: atomic `(tenant, device, key, operation)` reservation
  via unique index, replay-while-processing / replay-after-success /
  hash-conflict semantics per ADR-0002.
- **Client mutation policy**: no automatic mutation retries; the only
  network replay (401 → refresh → replay) reuses the same envelope, so it
  is idempotent. Correct for a POS.
- **Bundle discipline**: every route lazy, vendors chunked, perf budgets
  gated in CI (`perf-budget.json`, Lighthouse + Electron memory gates),
  i18n namespace parity tested.

## World-class gap analysis

What separates Puntovivo today from a world-class LatAm retail POS
platform, in dependency order:

1. **Fiscal provider abstraction (the sellability gate).** The fiscal
   foundations (document rows, mock provider, settings router) exist, but
   production sale is gated on a DIAN-authorized provider. Define a
   `FiscalProvider` interface (submit, poll, contingency, XML retention)
   now, implement the mock against it, and the first real provider
   (e.g. a Colombian PAC) becomes an adapter instead of a rewrite.
   Contingency mode (offline folio issuance + later transmission) should
   be modeled in the state machine from day one. Effort: L.
2. **Sync conflict resolution story.** The outbox/idempotency machinery is
   solid for at-least-once push, but multi-site + hub_client growth needs
   an explicit conflict policy per entity (last-write-wins with audit
   trail vs. merge for stock counters). Document per-table policies before
   cloud expansion; CRDT-style counters only where stock is concurrently
   mutated. Effort: M–L.
3. **Hardware abstraction layer.** ESC/POS transport policy is a strong
   start. Promote it to a peripherals HAL (printer, drawer, scanner,
   scale, payment terminal) with per-device capability descriptors and a
   certification matrix doc, so the physical-lab validation gate becomes a
   checklist instead of ad-hoc testing. Effort: M.
4. **Update/rollback strategy.** Differential updates exist; add staged
   rollout percentages and a one-click rollback channel (keep N-1
   installer cached locally) before fleet size makes a bad update
   expensive. Effort: M.
5. **Backup/restore UX.** The encrypted bundle + cross-device restore flow
   is technically excellent; make it operator-proof: scheduled automatic
   backups with retention, restore rehearsal prompt, and a recovery-key
   escrow printout at first run. Effort: M.
6. **Observability for operators, not just developers.** Crash telemetry
   and readiness probes exist. Add a fleet health view (per-site sync lag,
   pending outbox depth, last backup age, update version spread) — the
   Operations Center surface is the natural home. Effort: M.
7. **Licensing/activation.** Before hosted SaaS: per-tenant license
   issuance bound to the existing device registration, offline grace
   period, and a kill-switch policy that never bricks a register mid-shift
   (degrade to read-only + printing). Effort: M.
8. **Plugin/module system maturity.** Module gating exists client- and
   server-side; formalize the module manifest (permissions, migrations,
   surfaces) so verticals (restaurant, pharmacy) and third parties ship as
   packages instead of branches. Effort: L.
